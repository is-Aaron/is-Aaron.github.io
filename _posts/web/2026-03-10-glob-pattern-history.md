在日常的开发工作中，无论你是配置 `.gitignore` 忽略文件，还是在 Webpack 中打包资源，亦或是在终端里敲下一行带 `*` 的命令，你都在使用一项古老而强大的技术——**Glob 模式（Globbing）**。

它看似不起眼，却像空气一样无处不在。今天，让我们拨开历史的尘埃，沿着时间线，看看这几个简单的符号是如何从早期 Unix 系统的一段底层代码，演变成如今统治整个软件工程界的路径匹配标准的。

## 📍 1970s：创世之初——从 `/etc/glob` 说起

时间回到 20 世纪 70 年代，那是 Unix 操作系统的黎明时期。在早期的 Unix 版本（如 Unix V1 到 V6）中，命令行解释器（Shell）极其简陋，它甚至不知道如何处理通配符。

如果你在当时的终端里输入 `ls *.c`，Shell 自身是无法理解 `*.c` 是什么的。为了解决这个问题，Bell 实验室的 Unix 联合创始人肯·汤普逊（Ken Thompson）等人编写了一个独立的辅助程序，路径通常在 `/etc/glob`（`glob` 是 **global command** 的缩写）。

当 Shell 遇到包含特殊字符的命令时，它会在后台悄悄调用 `/etc/glob` 程序。这个程序会将 `*.c` 展开成 `main.c utils.c` 等实际的文件列表，然后再把这些列表传递给 `ls` 命令。

在这个**起源阶段**，Glob 奠定了它最核心的基础语法：
*   `*` (星号)：匹配零个或多个字符。
*   `?` (问号)：严格匹配任意单个字符。

## 📍 1970s末 - 1980s：Shell 的黄金时代与语法扩张

随着 1979 年 Unix V7 的发布（带来了著名的 Bourne Shell），以及随后的 C Shell (csh) 和 KornShell (ksh) 的普及，Shell 迎来了百花齐放的时代。开发者们意识到，频繁调用外部的 `/etc/glob` 程序太慢了。于是，Glob 模式被直接硬编码集成到了各家 Shell 的内部。

在这个**扩张阶段**，为了满足系统管理员日益复杂的日常需求，Glob 的词汇表得到了极大的丰富：
*   **字符类 `[...]`**：源自正则表达式的灵感，允许匹配特定范围的字符。比如 `rm file[0-9].txt` 可以精准删除 `file0.txt` 到 `file9.txt`。
*   **大括号扩展 `{...}`**：最初由 C shell (csh) 引入。它允许用户用逗号分隔多个后缀或前缀。比如 `*.{jpg,png}` 极大地简化了多类型文件的匹配逻辑。（*注：严格在计算机科学定义中，大括号扩展发生在 Globbing 路径匹配之前，属于纯文本的排列组合展开，但如今它早已被大众视作广义 Glob 语法中不可或缺的一部分*）。

## 📍 1990s - 2000s：跨越层级的革命——Globstar (`**`) 的诞生

随着软件项目的规模开始膨胀，目录结构变得越来越深。传统的 `*` 遇到了一个致命瓶颈：**它不能跨越目录边界（`/`）**。
如果你想找到项目中所有的 `.js` 文件，如果仅靠基础通配符，你可能得写出极其反人类的代码：`ls *.js */*.js */*/*.js`；或者只能被迫放弃纯 Glob 语法，转而调用繁琐的 `find . -name "*.js"` 命令。

在 1990 年代初，强大的 **zsh**（Z shell）率先站了出来，引入了一个革命性的符号：**双星号 `**` (递归通配符)**。
`**` 的出现彻底打破了目录层级的限制，它可以**递归地匹配零个或多个深度的子目录**。进入 21 世纪后，ksh93（创造了 "Globstar" 这一术语）和 Bash（在 2009 年的 4.0 版本中通过 `shopt -s globstar` 选项）也相继收编了这一特性。Glob 真正具备了无死角的“全局”搜索能力。

## 📍 2010s 至今：破圈而出——统治现代前端与工具链

如果 Glob 的故事只停留在终端里，那它只是一项优秀的系统工具。但随着 Node.js 和前端工程化的爆发，Glob 迎来了它的**现代纪元**。

开发者们发现，Glob 模式极其适合用来描述构建工具中的文件操作规则。于是，Glob 从 Shell 的专属技能，变成了跨平台的编程标准：
*   **版本控制**：Git 使用基于 Glob 的规则集作为 `.gitignore` 的标准。并且发扬了使用 `!` 作为前缀表示**取反/排除**的用法（例如 `!src/main.js` 表示不再忽略该文件）。
*   **Node.js 生态**：Isaac Z. Schlueter（npm 的创始人）编写了著名的 `glob` 库（早期曾被称为 `node-glob`）。
*   **工程化工具**：Webpack 的文件 Loader、ESLint 的校验规则、Prettier 的格式化范围、甚至 CI/CD 管道（如 GitHub Actions），全都在使用 Glob 模式。

如今，Glob 已经脱离了具体的操作系统，成为了一种通用的“文件系统查询语言”。

## 💻 命令行实战：感受现代 Glob 的威力

为了让你直观感受 Glob 发展至今的威力，我们来看一个结合了现代 Glob 特性的实际 Shell 命令。

```bash
tar -czvf backup_assets.tar.gz project/**/*.{jpg,png,svg}
```

根据命令解析规范，以下是该 Shell 命令的**全面解析**：

*   **主要用途与整体功能**：
    该命令的作用是递归扫描 `project` 目录及其所有子层级目录，找出所有的 `.jpg`、`.png` 和 `.svg` 图片文件，并将它们统统打包并压缩成一个名为 `backup_assets.tar.gz` 的归档文件。

*   **参数、选项和标志的详细分解**：
    *   `tar`：(Tape Archive) Unix/Linux 下标准的打包和压缩工具命令。
    *   `-c`：(create) 指示 tar 创建一个新的归档文件。
    *   `-z`：(gzip) 指示 tar 在打包的同时，调用 gzip 算法进行压缩，以减小文件体积。
    *   `-v`：(verbose) 冗长模式，在终端屏幕上逐一显示正在被打包的详细文件路径。
    *   `-f`：(file) 指定输出的归档文件名称。**注意：此标志后面必须紧跟文件名**。
    *   `backup_assets.tar.gz`：由 `-f` 指定的目标压缩包的文件名。
    *   `project/**/*.{jpg,png,svg}`：输入源路径，利用了 Glob 的高级匹配能力生成庞大的文件列表传递给 tar。

*   **特殊字符或符号的精确含义与功能**：
    *   `-` *(连字符)*：用于引导命令的短格式选项（如 `-c`, `-z`）。在 tar 中有时可省略，但加上是更好的规范。
    *   `.` *(点号)*：在 `backup_assets.tar.gz` 中，作为普通字符用于分隔文件名和扩展名；在 `{...}` 内部作为匹配后缀的一部分。
    *   `/` *(斜杠)*：Unix/Linux 系统下的标准目录层级分隔符。
    *   `**` *(双星号 / Globstar)*：**跨目录递归匹配符**。它的精确功能是匹配 `project/` 目录下的当前层级、子层级、孙子层级等**任意深度**的目录结构。
    *   `*` *(单星号)*：**基础通配符**。在这里精确匹配所在层级文件名中点号（`.`）之前的零个或多个任意字符。
    *   `{...}` *(大括号)*：**大括号扩展符 (Brace Expansion)**。它的精确功能是将前面的模式复制并展开，生成多个并列的匹配条件。
    *   `,` *(逗号)*：位于大括号内部，用作多个备选展开项的**精确分隔符**。整个 Glob 表达式最终会被 Shell 展开为匹配 `*.jpg`、`*.png` 或 `*.svg` 的所有文件集合。

## 结语

从 1970 年代一段简单的 C 语言代码，到今天无处不在的路径匹配标准，Glob 模式的历史完美诠释了 Unix 哲学：“做一件事，并把它做好”。下次当你敲下带 `*` 的命令，或是配置 `.gitignore` 时，不妨回味一下这段跨越了半个世纪的技术沉淀。
