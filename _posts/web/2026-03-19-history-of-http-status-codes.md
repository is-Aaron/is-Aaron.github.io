HTTP 响应状态码是我们每天浏览网页、使用 App 时都在幕后默默工作的“暗语”。它们的设计并非一蹴而就，而是随着万维网（World Wide Web）从简单的文档共享系统，演变成今天复杂的分布式应用平台而逐步发展起来的。

这套 **“三位数字”** 的设计实际上并非 HTTP 首创，而是借鉴了更早期的互联网协议（如 FTP 和 SMTP）。它的设计哲学非常精妙：**第一位数字代表了响应的类别**（方便机器和底层组件快速分类处理），**后两位数字提供具体细节**，而紧随其后的**原因短语（Reason Phrase，如 "OK" 或 "Not Found"）**则是给人类开发者看的。

我们可以从历史演进的角度，按五大类别来重新认识这些响应码：

## 🕰️ 史前时代：HTTP/0.9 (1991年)

在最早的 HTTP/0.9 版本中，**根本没有响应码的概念**。客户端发送一个 `GET /index.html`，服务器要么把 HTML 内容吐回来，要么直接断开 TCP 连接（代表出错了）。没有 Header，没有状态码，非常简陋。

## 🏗️ 奠基时代：HTTP/1.0 (1996年) 引入五大分类

随着网络变得复杂，HTTP/1.0（RFC 1945）正式引入了状态行（如 `HTTP/1.0 200 OK`）和状态码的五大基础分类：

### 🟢 2xx：成功类别 (Success) —— “服务器说：搞定了！”
这是最古老也最讨人喜欢的类别。
*   **200 OK**：最经典的响应，表示请求成功，数据在路上。
*   **201 Created**：后来随着 POST 和 PUT 方法被广泛用于创建资源（如 RESTful API），201 成为了专门表示“资源已成功创建”的标志。
*   **204 No Content**：请求成功了，但服务器没有什么内容要返回给你（比如你提交了一个表单，服务器只需要告诉你收到了，不需要刷新页面）。

### 🔴 4xx：客户端错误 (Client Error) —— “服务器说：这是你的锅！”
当客户端发来的请求有语法错误，或者请求了不存在的东西时使用。
*   **400 Bad Request**：客户端发送的请求语法有问题（比如少传了必填参数）。
*   **401 Unauthorized**：未授权。注意，这里的语义其实是**未认证（Unauthenticated）**，即你没有提供有效的身份凭证（没登录）。
*   **403 Forbidden**：禁止访问。这才是真正的**未授权**——服务器知道你是谁，但你权限不够。
*   **404 Not Found**：互联网上最著名的数字，你请求的资源（网页、图片）在服务器上找不到。

### ⚫ 5xx：服务器错误 (Server Error) —— “服务器说：我的锅，我崩了！”
*   **500 Internal Server Error**：最常见的服务器通用错误，通常意味着后端的代码抛出了未捕获的异常（比如 Java 空指针，Python 语法错误等）。
*   **502 Bad Gateway**（HTTP/1.0 引入）和 **504 Gateway Timeout**（HTTP/1.1 引入）：这两个状态码随着互联网架构的演进（反向代理、负载均衡、微服务）变得越来越常见。当你的 Nginx 或网关服务器尝试去请求后端的真实服务，但后端服务挂了（502）或者处理太慢超时了（504）时返回。

## 🚀 演进与修补时代：HTTP/1.1 (1997年) 及之后

HTTP/1.1（RFC 2068 及后续版本）极大地扩展了状态码，以支持**持久连接、缓存控制、断点续传**等复杂功能。

### ⚪ 1xx：信息性响应 (Informational) —— “服务器说：收到了一半，继续吧”
这个类别虽然在 HTTP/1.0 的标准中被预留，但当时并没有定义任何具体的 1xx 状态码（被标记为实验性质），它实际上是为了解决 HTTP/1.1 中的新问题才真正被使用起来的。
*   **100 Continue**：这是为了**节省带宽**发明的。如果客户端要上传一个超大文件（比如 10GB），它怕服务器拒绝，就会先发一个带 `Expect: 100-continue` 的请求头问问服务器：“我能传吗？” 服务器如果觉得没问题，就回一个 `100 Continue`，客户端这才开始传真正的文件体。
*   **101 Switching Protocols**：后来为了支持 **WebSocket** 这种长连接，客户端会请求升级协议，服务器如果同意，就会返回 101，然后这条 HTTP 连接就变成了 WebSocket 连接。

### 🟡 3xx：重定向 (Redirection) —— 一段“混乱”的演化史
3xx 家族的历史最能体现 HTTP 协议在现实世界中是如何“将错就错”并自我修复的。虽然 301、302 和 304 在 HTTP/1.0 时期就已经存在，但它们在 HTTP/1.1 中经历了重要的语义修补。

*   **301 Moved Permanently**（永久重定向）和 **304 Not Modified**（未修改）：304 是**现代 Web 缓存的基石**。浏览器问服务器：“这个文件自昨天起有修改吗？” 服务器说：“没有（304），你直接用本地缓存吧”，这样就省去了传输大文件的流量。
*   **⚔️ 302 的混乱与修复 (302 vs 303 vs 307)**：
    *   最初在 HTTP/1.0 中，**302 Found**（当时叫 Moved Temporarily）表示临时重定向。按照规范，如果客户端用 POST 方法提交表单后收到 302，它应该**继续用 POST** 请求新地址。
    *   **但是！** 早期的浏览器（如 IE、网景）全都不按套路出牌，它们收到 302 后，不管三七二十一，**统统改用 GET 方法**去请求新地址。这个错误实在太普及了，以至于成了事实上的标准。
    *   为了收拾这个烂摊子，HTTP/1.1 引入了两个新状态码来明确语义：
        *   **303 See Other**：明确规定客户端必须把方法**变成 GET** 去请求新地址（这就是现在网页表单提交后防止刷新重复提交的 PRG 模式标准做法）。
        *   **307 Temporary Redirect**：严格版本的 302，明确规定客户端**绝对不准改变请求方法**（POST 必须还是 POST）。
        *   后来又加入了 **308 Permanent Redirect**，作为 301 的严格版本（不准改变方法）。

## 🌟 现代 API 时代的补充 (RFC 6585, RFC 7725 等)

随着 RESTful API 和现代云原生架构的兴起，原有的状态码不够用了，于是 IETF 陆续补充了一些非常有时代特征的状态码：

*   **422 Unprocessable Entity**：虽然请求格式对，但内容有逻辑错误。现在被广泛用于 **API 表单参数校验失败**（比如邮箱格式不对、密码太短）。（最初来源于 WebDAV 协议）。
*   **429 Too Many Requests** (2012年加入)：现代 API 必备。防止恶意爬虫或 DDoS 攻击，告诉客户端：**“你请求太快了，被限流了，歇会儿吧。”**
*   **451 Unavailable For Legal Reasons** (2016年加入，RFC 7725)：非常有赛博朋克感的一个状态码。用于表示内容因法律原因（如版权审查、政府要求）被封禁。这个数字是在向反乌托邦小说《华氏 451 度》（Fahrenheit 451，讲述焚书的故事）致敬。
*   **☕ 418 I'm a teapot**：这是一个著名的互联网彩蛋。诞生于 1998 年愚人节的超文本咖啡壶控制协议（RFC 2324）。“我是一个茶壶，所以拒绝冲泡咖啡”。现在很多框架（如 Node.js, Spring）和极客网站仍支持并保留了这个彩蛋。

## 总结

从最早只有 HTML 文本，到今天支撑整个微服务、音视频流媒体和复杂业务逻辑的 API 生态，HTTP 状态码的演变史，其实就是一部**万维网（WWW）从简单静态文档系统进化为复杂分布式软件系统的历史**。理解了它们的历史和语义，能够帮助我们设计出更优雅、更符合规范的现代 Web 系统。
